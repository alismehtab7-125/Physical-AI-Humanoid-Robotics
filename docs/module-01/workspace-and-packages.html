Setting Up Workspace and Packages

ROS 2 Colcon Workspace​

A workspace in ROS 2 is a directory where you modify, build, and install ROS 2 packages. The standard build tool for ROS 2 is Colcon, which replaces the older catkin tools from ROS 1.

Creating a Colcon Workspace​

Step 1: Create Workspace Directory​

First, create a directory for your workspace:

mkdir -p ~/ros2_ws/src
cd ~/ros2_ws

This creates a workspace named ros2_ws with a src directory where you'll place your source code.

Step 2: Source ROS 2 Environment​

Before building, make sure ROS 2 is sourced:

source /opt/ros/humble/setup.bash  # For Ubuntu with ROS 2 Humble
# OR for other distributions:
# source /opt/ros/iron/setup.bash   # Iron
# source /opt/ros/rolling/setup.bash # Rolling

On Windows:

call "C:\dev\ros_humble\setup.bat"

Step 3: Build the Workspace​

Even though the src directory is empty, you can build the workspace:

colcon build

This will create three directories:

  • build/: Contains intermediate build files
  • install/: Contains the installed packages (this is what you'll source)
  • log/: Contains build logs

Step 4: Source the Workspace​

After building, source the workspace to use your packages:

source ~/ros2_ws/install/setup.bash

To make this permanent, add it to your .bashrc:

echo "source ~/ros2_ws/install/setup.bash" >> ~/.bashrc

Creating a Python Package​

Using ros2 pkg Command​

To create a new Python package, use the ros2 pkg create command:

cd ~/ros2_ws/src
ros2 pkg create --build-type ament_python my_robot_package

This creates a basic Python package structure:

my_robot_package/
├── my_robot_package/
│   ├── __init__.py
│   └── my_node.py
├── setup.py
├── setup.cfg
├── package.xml
└── resource/
    └── my_robot_package

Package Structure Explained​

  • my_robot_package/: The main Python module directory
  • __init__.py: Makes it a Python package
  • my_node.py: A sample Python node file
  • setup.py: Python package setup file
  • setup.cfg: Configuration for installation
  • package.xml: Package manifest with dependencies and metadata
  • resource/: Contains package resources

Customizing package.xml​

Edit the package.xml file to add dependencies and metadata:

<?xml version="1.0"?>
<?xml-model href="http://download.ros.org/schema/package_format3.xsd" schematypens="http://www.w3.org/2001/XMLSchema"?>
<package format="3">
  <name>my_robot_package</name>
  <version>0.0.0</version>
  <description>Example ROS 2 Python package</description>
  <maintainer email="user@example.com">Your Name</maintainer>
  <license>Apache-2.0</license>

  <depend>rclpy</depend>
  <depend>std_msgs</depend>

  <test_depend>ament_copyright</test_depend>
  <test_depend>ament_flake8</test_depend>
  <test_depend>ament_pep257</test_depend>
  <test_depend>python3-pytest</test_depend>

  <export>
    <build_type>ament_python</build_type>
  </export>
</package>

Creating a Python Node​

Basic Node Structure​

Create or modify my_robot_package/my_node.py:

#!/usr/bin/env python3

import rclpy
from rclpy.node import Node
from std_msgs.msg import String


class MyNode(Node):
    def __init__(self):
        super().__init__('my_node')
        self.publisher = self.create_publisher(String, 'my_topic', 10)
        self.subscription = self.create_subscription(
            String,
            'my_input_topic',
            self.listener_callback,
            10
        )
        self.timer = self.create_timer(0.5, self.timer_callback)
        self.i = 0

    def timer_callback(self):
        msg = String()
        msg.data = f'Hello from Python: {self.i}'
        self.publisher.publish(msg)
        self.get_logger().info(f'Publishing: "{msg.data}"')
        self.i += 1

    def listener_callback(self, msg):
        self.get_logger().info(f'I heard: "{msg.data}"')


def main(args=None):
    rclpy.init(args=args)
    node = MyNode()
    rclpy.spin(node)
    node.destroy_node()
    rclpy.shutdown()


if __name__ == '__main__':
    main()

Making the Node Executable​

Make the node file executable:

chmod +x my_robot_package/my_node.py

Building and Running​

Building the Package​

From the workspace root:

cd ~/ros2_ws
colcon build --packages-select my_robot_package

Or build all packages:

cd ~/ros2_ws
colcon build

Sourcing and Running​

source ~/ros2_ws/install/setup.bash
ros2 run my_robot_package my_node

Adding Executables to setup.py​

To make your nodes available as ROS 2 executables, edit setup.py:

from setuptools import setup
import os
from glob import glob

package_name = 'my_robot_package'

setup(
    name=package_name,
    version='0.0.0',
    packages=[package_name],
    data_files=[
        ('share/ament_index/resource_index/packages',
            ['resource/' + package_name]),
        ('share/' + package_name, ['package.xml']),
        (os.path.join('share', package_name), glob('launch/*launch.[pxy][yma]*')),
    ],
    install_requires=['setuptools'],
    zip_safe=True,
    maintainer='Your Name',
    maintainer_email='you@example.com',
    description='Example ROS 2 Python package',
    license='Apache-2.0',
    tests_require=['pytest'],
    entry_points={
        'console_scripts': [
            'my_node = my_robot_package.my_node:main',
        ],
    },
)

Useful Colcon Commands​

Build Specific Package​

colcon build --packages-select my_robot_package

Build with Symlink Install (for faster rebuilds during development)​

colcon build --symlink-install

Run Tests​

colcon test
colcon test-result --all

Clean Build​

rm -rf build/ install/ log/

Best Practices​

  1. Use meaningful package names: Follow ROS naming conventions (lowercase with underscores)
  2. Organize your code: Group related functionality in logical packages
  3. Document dependencies: List all required dependencies in package.xml
  4. Use virtual environments: For development, consider using Python virtual environments
  5. Test your packages: Write unit tests and integration tests
  6. Follow coding standards: Use linters like flake8 for Python code

Troubleshooting​

Common Issues:​

  1. Package not found: Make sure to source the workspace before running
  2. Import errors: Check that dependencies are listed in package.xml
  3. Build errors: Verify that all files have correct permissions and syntax
