Integrating Models and ROS 2 Plugins in Gazebo

Gazebo Model Format and Structure​

SDF (Simulation Description Format)​

Gazebo uses the Simulation Description Format (SDF) as its primary model description language. SDF is an XML-based format that describes:

  • Worlds: Complete simulation environments with objects, lighting, and physics
  • Models: Individual robot or object descriptions
  • Actuators: Joint controllers and mechanisms
  • Sensors: Camera, LIDAR, IMU, and other sensor configurations

Basic SDF Structure​

<?xml version="1.0" ?>
<sdf version="1.7">
  <model name="my_robot">
    <pose>0 0 0.5 0 0 0</pose>
    <link name="chassis">
      <pose>0 0 0.1 0 0 0</pose>
      <collision name="collision">
        <geometry>
          <box>
            <size>1.0 0.5 0.2</size>
          </box>
        </geometry>
      </collision>
      <visual name="visual">
        <geometry>
          <box>
            <size>1.0 0.5 0.2</size>
          </box>
        </geometry>
      </visual>
      <inertial>
        <mass>10.0</mass>
        <inertia>
          <ixx>0.416</ixx>
          <ixy>0.0</ixy>
          <ixz>0.0</ixz>
          <iyy>1.04</iyy>
          <iyz>0.0</iyz>
          <izz>1.04</izz>
        </inertia>
      </inertial>
    </link>
  </model>
</sdf>

World Files​

Creating World Files​

World files define the complete simulation environment, including:

  • Static objects and obstacles
  • Lighting and atmospheric conditions
  • Physics engine parameters
  • Initial robot placements
<?xml version="1.0" ?>
<sdf version="1.7">
  <world name="my_world">
    <!-- Physics engine configuration -->
    <physics name="default_physics" type="ode">
      <max_step_size>0.001</max_step_size>
      <real_time_factor>1.0</real_time_factor>
      <real_time_update_rate>1000.0</real_time_update_rate>
    </physics>

    <!-- Lighting -->
    <light name="sun" type="directional">
      <cast_shadows>true</cast_shadows>
      <pose>0 0 10 0 0 0</pose>
      <diffuse>0.8 0.8 0.8 1</diffuse>
      <specular>0.2 0.2 0.2 1</specular>
      <attenuation>
        <range>1000</range>
        <constant>0.9</constant>
        <linear>0.01</linear>
        <quadratic>0.001</quadratic>
      </attenuation>
      <direction>-0.6 0.4 -0.8</direction>
    </light>

    <!-- Ground plane -->
    <include>
      <uri>model://ground_plane</uri>
    </include>

    <!-- Your custom models -->
    <include>
      <uri>model://my_robot</uri>
      <pose>0 0 0.5 0 0 0</pose>
    </include>
  </world>
</sdf>

Integrating Custom Robot Models​

Model Directory Structure​

A typical Gazebo model follows this structure:

~/.gazebo/models/my_robot/
├── model.config
├── model.sdf
├── meshes/
│   ├── chassis.dae
│   ├── wheel.dae
│   └── ...
└── materials/
    ├── textures/
    └── scripts/

Model Configuration File​

The model.config file provides metadata about the model:

<?xml version="1.0"?>
<model>
  <name>My Robot</name>
  <version>1.0</version>
  <sdf version="1.7">model.sdf</sdf>
  <author>
    <name>Your Name</name>
    <email>your.email@example.com</email>
  </author>
  <description>A sample robot model for Gazebo simulation.</description>
</model>

Advanced Model Features​

<model name="my_robot">
  <!-- Self-collide enables collision checking between links of the same model -->
  <self_collide>false</self_collide>

  <!-- Static models don't move and have no physics -->
  <static>false</static>

  <!-- Kinematic models follow kinematic equations rather than dynamic simulation -->
  <kinematic>false</kinematic>

  <!-- Enable wind effects on this model -->
  <enable_wind>false</enable_wind>

  <link name="chassis">
    <!-- Turn off gravity for this link -->
    <gravity>true</gravity>

    <!-- Self-collide for this specific link -->
    <self_collide>false</self_collide>

    <!-- Enable/disable the link -->
    <enable_wind>false</enable_wind>

    <!-- Visual properties -->
    <visual name="chassis_visual">
      <geometry>
        <mesh>
          <uri>model://my_robot/meshes/chassis.dae</uri>
        </mesh>
      </geometry>
      <material>
        <ambient>0.1 0.1 0.1 1</ambient>
        <diffuse>0.7 0.7 0.7 1</diffuse>
        <specular>0.01 0.01 0.01 1</specular>
      </material>
    </visual>

    <!-- Collision properties -->
    <collision name="chassis_collision">
      <geometry>
        <mesh>
          <uri>model://my_robot/meshes/chassis_collision.dae</uri>
        </mesh>
      </geometry>
      <surface>
        <friction>
          <ode>
            <mu>1.0</mu>
            <mu2>1.0</mu2>
          </ode>
        </friction>
      </surface>
    </collision>
  </link>

  <!-- Joint between links -->
  <joint name="wheel_joint" type="revolute">
    <parent>chassis</parent>
    <child>wheel</child>
    <axis>
      <xyz>0 1 0</xyz>
      <limit>
        <lower>-1.5707</lower>
        <upper>1.5707</upper>
        <effort>100</effort>
        <velocity>10</velocity>
      </limit>
    </axis>
  </joint>
</model>

ROS 2 Gazebo Plugins​

Overview of ROS 2 Integration​

ROS 2 integration in Gazebo is achieved through plugins that bridge the gap between Gazebo's native communication system and ROS 2 topics and services. These plugins enable:

  • Publishing sensor data to ROS 2 topics
  • Controlling robot joints via ROS 2 messages
  • Managing robot state through ROS 2 services

Common ROS 2 Plugins​

1. Differential Drive Plugin​

The differential drive plugin provides a complete solution for controlling wheeled robots:

<plugin name="diff_drive" filename="libgazebo_ros_diff_drive.so">
  <left_joint>left_wheel_joint</left_joint>
  <right_joint>right_wheel_joint</right_joint>
  <wheel_separation>0.3</wheel_separation>
  <wheel_diameter>0.1</wheel_diameter>
  <max_wheel_torque>20</max_wheel_torque>
  <max_wheel_acceleration>1.0</max_wheel_acceleration>
  <command_topic>cmd_vel</command_topic>
  <odometry_topic>odom</odometry_topic>
  <odometry_frame>odom</odometry_frame>
  <robot_base_frame>base_footprint</robot_base_frame>
  <publish_odom>true</publish_odom>
  <publish_wheel_tf>false</publish_wheel_tf>
  <publish_odom_tf>true</publish_odom_tf>
  <ros>
    <namespace>/my_robot</namespace>
  </ros>
</plugin>

2. Joint State Publisher Plugin​

This plugin publishes joint states to the /joint_states topic:

<plugin name="joint_state_publisher" filename="libgazebo_ros_joint_state_publisher.so">
  <ros>
    <namespace>/my_robot</namespace>
    <remapping>~/out:=joint_states</remapping>
  </ros>
  <update_rate>30</update_rate>
  <joint_name>left_wheel_joint</joint_name>
  <joint_name>right_wheel_joint</joint_name>
</plugin>

3. Camera Sensor Plugin​

Camera sensors publish image data to ROS 2 topics:

<sensor name="camera" type="camera">
  <camera name="head">
    <horizontal_fov>1.3962634</horizontal_fov>
    <image>
      <width>800</width>
      <height>600</height>
      <format>R8G8B8</format>
    </image>
    <clip>
      <near>0.1</near>
      <far>100</far>
    </clip>
  </camera>
  <plugin name="camera_controller" filename="libgazebo_ros_camera.so">
    <ros>
      <namespace>/my_robot</namespace>
      <remapping>~/image_raw:=/camera/image_raw</remapping>
      <remapping>~/camera_info:=/camera/camera_info</remapping>
    </ros>
    <camera_name>camera</camera_name>
    <frame_name>camera_frame</frame_name>
    <hack_baseline>0.07</hack_baseline>
    <distortion_k1>0.0</distortion_k1>
    <distortion_k2>0.0</distortion_k2>
    <distortion_k3>0.0</distortion_k3>
    <distortion_t1>0.0</distortion_t1>
    <distortion_t2>0.0</distortion_t2>
  </plugin>
</sensor>

4. IMU Sensor Plugin​

IMU sensors provide inertial measurement data:

<sensor name="imu_sensor" type="imu">
  <always_on>true</always_on>
  <update_rate>100</update_rate>
  <visualize>true</visualize>
  <topic>__default_topic__</topic>
  <plugin name="imu_plugin" filename="libgazebo_ros_imu_sensor.so">
    <ros>
      <namespace>/my_robot</namespace>
      <remapping>~/out:=imu</remapping>
    </ros>
    <initial_orientation_as_reference>false</initial_orientation_as_reference>
    <frame_name>imu_link</frame_name>
  </plugin>
</sensor>

Creating Custom Plugins​

Basic Plugin Structure​

To create a custom Gazebo plugin, you need to inherit from the appropriate base class:

#include <gazebo/gazebo.hh>
#include <gazebo/physics/physics.hh>
#include <gazebo/transport/transport.hh>
#include <gazebo/msgs/msgs.hh>

#include <thread>

namespace gazebo
{
  class CustomPlugin : public ModelPlugin
  {
    public: void Load(physics::ModelPtr _parent, sdf::ElementPtr /*_sdf*/)
    {
      // Store the model pointer for convenience
      this->model = _parent;

      // Listen to the update event. This event is broadcast every
      // simulation iteration.
      this->updateConnection = event::Events::ConnectWorldUpdateBegin(
          std::bind(&CustomPlugin::OnUpdate, this));
    }

    // Called by the world update start event
    public: void OnUpdate()
    {
      // Do something every update
    }

    // Pointer to the model
    private: physics::ModelPtr model;

    // Pointer to the update event connection
    private: event::ConnectionPtr updateConnection;
  };

  // Register this plugin with the simulator
  GZ_REGISTER_MODEL_PLUGIN(CustomPlugin)
}

CMakeLists.txt for Custom Plugins​

cmake_minimum_required(VERSION 3.8)
project(my_gazebo_plugins)

if(CMAKE_COMPILER_IS_GNUCXX OR CMAKE_CXX_COMPILER_ID MATCHES "Clang")
  add_compile_options(-Wall -Wextra -Wpedantic)
endif()

# Find packages
find_package(ament_cmake REQUIRED)
find_package(gazebo REQUIRED)
find_package(gazebo_dev REQUIRED)

# Include directories
include_directories(
  include
  ${GAZEBO_INCLUDE_DIRS}
)

# Link directories
link_directories(
  ${GAZEBO_LIBRARY_DIRS}
)

# Add plugin library
add_library(my_custom_plugin SHARED
  src/my_custom_plugin.cpp
)

# Link libraries
target_link_libraries(my_custom_plugin
  ${GAZEBO_LIBRARIES}
)

# Install plugin
install(TARGETS my_custom_plugin
  LIBRARY DESTINATION lib
)

ament_package()

Best Practices for Model and Plugin Integration​

1. Model Organization​

  • Use descriptive names for models, links, and joints
  • Organize models in a clear directory structure
  • Include proper metadata in model.config files
  • Use consistent coordinate frames and units

2. Physics Parameters​

  • Set realistic mass and inertia values
  • Configure appropriate friction coefficients
  • Use collision-optimized meshes separate from visual meshes
  • Validate physics parameters against real hardware when possible

3. Sensor Configuration​

  • Match sensor parameters to real hardware specifications
  • Add appropriate noise models to simulate real sensor behavior
  • Configure proper update rates for each sensor type
  • Set appropriate ranges and resolutions

4. Plugin Configuration​

  • Use appropriate namespaces to avoid topic conflicts
  • Configure update rates based on computational requirements
  • Test plugins thoroughly in isolation before integration
  • Document plugin parameters and expected behavior

5. Performance Optimization​

  • Simplify collision meshes for better performance
  • Use appropriate update rates (not all sensors need maximum frequency)
  • Limit the number of active physics objects in the scene
  • Use static models when possible instead of dynamic ones

Troubleshooting Common Issues​

1. Model Not Loading​

  • Check that the model directory is in Gazebo's model path
  • Verify the model.config file exists and is properly formatted
  • Ensure all referenced mesh files exist and are accessible

2. Plugin Not Working​

  • Check that the plugin library is properly installed and linked
  • Verify plugin filename matches the compiled library
  • Check ROS 2 topic names and permissions
  • Review Gazebo console output for error messages

3. Physics Issues​

  • Verify mass and inertia values are physically realistic
  • Check joint limits and constraints
  • Adjust physics update rates if needed
  • Ensure proper coordinate frame conventions

By following these guidelines and best practices, you can successfully integrate custom robot models and ROS 2 plugins into Gazebo, creating realistic and functional simulation environments for your robotic applications.
